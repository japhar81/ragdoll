# ADR 0022 — Connect-RPC as the external-plugin transport

## Status

Accepted (Phase A landed: 2026-06-01).
Phase B follow-ups tracked separately (Python sidecar migration, helm
`appProtocol: http2`, OpenShift Route h2 annotations, cross-language
interop tests).

## Context

Before this change the external-plugin transport had two declared modes
and one implementation:

```ts
interface ExternalPluginEndpoint {
  mode: "http" | "grpc";
  baseUrl: string;
  healthPath?: string;
  executePath?: string;
  timeoutMs?: number;
}
```

`mode: "http"` was a hand-rolled fetch POST to `/execute` with a JSON
body shaped by `buildExternalRequestBody()`. `mode: "grpc"` was a
literal `throw new Error("grpc external transport not implemented")` —
declared in the manifest enum, recognized by validation, never callable.

We also had an unwired streaming gap: `POST /api/pipelines/:id/stream`
accepted an `onToken` callback and threaded it through `runSyncPipeline`
into plugin execution, but the LLM plugins (`provider_chat`) checked
`provider.streamChat` which **no provider implemented**, so the streaming
branch was dead code and the SSE route only ever delivered lifecycle
frames.

A customer mention of "we'd like real token streaming" + an internal
question on whether to commit to gRPC for at-scale plugin streaming
forced the decision: pick the streaming transport story end-to-end now,
not later.

## Options considered

We laid out three:

1. **NDJSON-over-chunked-HTTP only.** No new deps; symmetrical with the
   SSE browser path. Ceiling: TCP-only backpressure; weak typing across
   the plugin boundary.
2. **gRPC only (unary + streaming).** One wire, type-safe, real
   backpressure, high author-onboarding floor (every plugin author needs
   `protoc`), HTTP/2-end-to-end deployment story.
3. **HTTP request/response + gRPC streaming** ("do both"). Two transports
   first-class, sized to each shape's strength. Maintenance tax on
   shared concerns implemented twice.

The user picked #3 with full bidi included, with a hard requirement that
the "two transports forever" tax be paid for via discipline (one .proto
as truth, one internal runtime type per shape, shared concerns layer
implemented once, one author SDK per language).

A spike (kept in `.claude/worktrees/spike-grpc-http-generator/spike-grpc/`
during development) validated all five "clean #3" invariants under
connect-rpc — one Connect-Node server handler answers Connect HTTP/JSON,
native gRPC, and gRPC-Web simultaneously from the same router.

## Decision

**Adopt [connect-rpc](https://connectrpc.com/) (`@connectrpc/connect` +
`@connectrpc/connect-node` on Node, `connectrpc` on PyPI) as the
external-plugin transport, with `proto/plugin.proto` as the single
source of truth.**

The new `ExternalPluginEndpoint` shape:

```ts
interface ExternalPluginEndpoint {
  baseUrl: string;
  protocol?: "connect" | "grpc" | "grpc-web"; // default "connect"
  httpVersion?: "1.1" | "2";                  // default 1.1; "2" required for "grpc"
  timeoutMs?: number;
}
```

`mode`, `healthPath`, and `executePath` are gone — Connect uses fixed RPC
routes derived from the proto package (`/ragdoll.plugin.v1.PluginRuntime/*`).
`protocol` is a *client preference*: the server always answers all three
protocols from the same handler, so picking `grpc` for backpressure-heavy
streaming and `connect` for everything else is a per-call switch, not a
deployment commitment.

## Why connect-rpc

- **One handler, three protocols.** Connect-Node serves Connect HTTP/JSON
  + gRPC-Web + native gRPC from the same listener — no reverse proxy
  (avoiding the grpc-gateway Go-process tax). Plugin authors write one
  function; the runtime picks the wire.
- **All four RPC kinds supported.** Unary, server-stream, client-stream,
  bidi — defined uniformly in the `.proto`, dispatched through the same
  router.
- **TypeScript-first** but mature in Python too. `connectrpc` on PyPI
  passes the official Connect conformance suite and works with ASGI
  (Starlette / FastAPI / Hypercorn).
- **`buf generate` already exists**: protoc-gen-es produces clean TS
  message and service descriptors; no Go toolchain required.
- **Curl-friendly for unary.** A plain `POST /ragdoll.plugin.v1.PluginRuntime/Execute`
  with `Content-Type: application/json` works — useful for support /
  debugging.

## Invariants we hold

These five must stay load-bearing for the "two transports stay clean"
property to hold over time. Reviewers should reject changes that violate
any of them:

1. **`proto/plugin.proto` is the single source of truth.** All TypeScript
   types come from `@ragdoll/proto-gen` (regenerated by `npm run generate`).
   No hand-authored equivalent of the wire shape in zod or anywhere else.
2. **One internal runtime type per shape.** The transport adapter is the
   only code that knows the wire. `executeExternalConnect()` returns
   `PluginExecutionOutput`; `executeExternalConnectStream()` returns the
   same after consuming the chunk stream. Everything downstream — the
   DAG executor, the SSE forwarder, the OTel spans — treats both modes
   identically.
3. **Shared concerns implemented once.** Auth, tenant scoping, allow-listed
   hosts, deny-private-networks, retry policy, OTel context propagation
   live in `defaultInterceptors()` (Phase A: empty placeholder; Phase B
   fills it). Both adapter call sites use the same composition.
4. **One author SDK per language.** `@ragdoll/plugin-sdk/author` exports
   `createPluginServer({pluginId, version, handlers})` — author writes
   one handler function; SDK exposes it over all three protocols. Per-
   method behaviour declared in the manifest (`manifest.streaming`), not
   per-plugin. Python equivalent (`ragdoll-plugin-py`) lands in Phase B
   with identical surface area.
5. **Parameterized test harness.** Every shared-concern test runs the
   same assertion against every (transport × RPC-kind) pair that's
   eligible. Adding a new concern forces a single test addition that
   then covers all eligible combinations.

## What landed in Phase A

- `proto/plugin.proto` defining `PluginRuntime` with `Health` + unary
  `Execute` + three streaming methods.
- `@ragdoll/proto-gen` workspace exporting generated TS descriptors;
  consumed by plugin-sdk + the author SDK.
- `executeExternalConnect` (unary, retried 3× with 250/750/2250ms
  backoff) and `executeExternalConnectStream` (server-stream, not
  retried — partial output is irrecoverable) replace the old
  `executeExternalHttp` + the gRPC `throw`.
- `PluginManifest.streaming?: boolean` declares streaming intent —
  runtime routes to the streaming RPC only when the manifest opts in
  AND the caller supplied `onToken`.
- `streamChat` implementations on `OpenAIProvider`, `AnthropicProvider`,
  and `OllamaCompatibleProvider` — closing the "SSE route never emits
  tokens" gap. SSE token streaming now works end-to-end.
- `@ragdoll/plugin-sdk/author` (`createPluginServer`) — thin
  one-call-server wrapper for future Node-side external plugin authors.
- Tests rewritten to exercise the actual Connect wire (real
  `connectNodeAdapter` server, real `createClient` against it) instead
  of stubbing the JSON body shape.

## Deferred to Phase B

- Migrate `services/python-plugins` (crawl4ai sidecar) to dual-host
  Connect alongside FastAPI; swap uvicorn → Hypercorn for HTTP/2 + true
  full-duplex bidi.
- Ship `ragdoll-plugin-py` (Python author SDK, identical surface to the
  Node one).
- Helm chart updates: `appProtocol: http2` on every Service fronting a
  gRPC-native pod (api if it ever serves callbacks; python-plugins);
  OpenShift Route annotations for h2 termination; values.yaml toggle for
  protocol selection.
- Cross-language interop integration tests (Node runtime → Python plugin)
  in `tests/e2e/`.
- Populate `defaultInterceptors()` — auth (bearer token), tenant scope
  check, allow-listed hosts enforcement, OTel traceparent extractor.

## Gotchas we accept

- **Full-duplex bidi requires HTTP/2 end-to-end.** Over HTTP/1.1 it's
  half-duplex (client must finish sending before server starts). uvicorn
  doesn't do HTTP/2; switch to Hypercorn when the Python sidecar needs
  bidi (Phase B). Kubernetes Services need `appProtocol: http2` or
  kube-proxy silently downgrades; OpenShift Routes need h2 annotations
  to terminate gRPC. None blockers; all documented.
- **Streaming requires the SDK on the client.** Connect's framing for
  streams uses a 5-byte length-prefix envelope per message — raw curl
  can hit unary but not streaming. Plugin docs make this explicit.
- **Intel Mac dev needs Rust.** The PyPI `pyqwest` (a `connectrpc`
  dependency) publishes wheels for macOS arm64 + Linux x86_64/aarch64
  but NOT macOS x86_64. Intel Mac developers must install a Rust
  toolchain (`brew install rust` or `rustup`) to compile from source.
  Production (`python:3.12-slim`) is Linux x86_64 → installs cleanly
  from wheels with no Rust.
