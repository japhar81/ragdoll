# ADR 0020: External-Database Plugins (Postgres family)

## Status

Accepted. **Superseded by ADR-0023 + ADR-0024.**

The architectural rules in this doc (SQL-as-config, params-as-data,
connections-as-secrets, no host/port/URL in plugin config) are still
the contract. What changed:

- The per-tenant `datasource_connections` registry this ADR introduced
  has been collapsed into the **unified `connections` table** (ADR-0023
  §1). Slugs preserved; cascade resolution (env → tenant → global)
  unchanged.
- Datasets no longer reference backends via `backends.<modality>.connectionName`;
  the binding shape is `bindings.<name>.{connection, collection?, namespace?}`
  per ADR-0023 §2.
- The `postgres` driver itself is now a `ConnectionDriverPlugin`
  (`postgresConnectionDriver` in `plugins/builtin-rag/src/postgres-core.ts`)
  picked up by the loader's module scan per ADR-0024. The imperative
  `registerConnectionDriver()` shim still exists for legacy tests but
  is no longer the registration path drivers ship through.
- The `postgres_query` / `postgres_upsert` / `postgres_delete` /
  `postgres_exec` plugins are unchanged in their public contract; they
  resolve their pooled client through `acquireClient(input.connection)`
  the same way every other family in the registry does.

Read this ADR for the per-plugin rules; read ADR-0023 + ADR-0024 for
the registry + driver-loading architecture that everything is built on
today.

## Context

Until now every storage-touching plugin in the platform talked to a
backend that the platform itself provisioned and managed: Qdrant for
vectors, OpenSearch for lexical / hybrid, Dgraph for graph. The
control-plane database (the one referenced by `DATABASE_URL`) is for
*Ragdoll's own orchestration state* — pipeline specs, plugin registry,
dataset metadata, run history, RBAC. ADR 0001 was explicit about the
boundary.

Several emerging use cases need pipelines to read from and write to an
external operational database the tenant already owns:

  - Pulling a small structured fact set (e.g. "the last 3 outages for
    project X") and rendering it into a prompt context.
  - Refreshing a status row after an LLM-driven status check.
  - Bulk-ingesting per-message rows during an email/event ingest run.
  - Running one-shot migrations to provision tables before an ingest
    pipeline first runs.

The choices were: (a) shove the data into Ragdoll's own DB, (b) write
ad-hoc per-tenant Python adapters, or (c) admit external databases as a
first-class plugin category. (a) breaks ADR 0001 and couples ingest
volume to control-plane DB capacity. (b) reinvents connection pooling,
secret handling, and identifier safety per adapter. (c) is what this ADR
proposes.

## Decision

Ship a Postgres plugin family — `postgres_query`, `postgres_upsert`,
`postgres_exec` — as the reference implementation of an
**external-database plugin** category, with four invariants every
external-DB plugin must satisfy:

### 1. Domain data lives outside Ragdoll's database

The external database is provisioned by the operator and exists
independently of Ragdoll. Ragdoll's DB stays focused on its mission:
orchestration state. Email facts, application events, business records,
etc. live in the external DB. Pipelines reach in via these plugins and
never via Ragdoll's own pool.

### 2. SQL is config, params are data

Every SQL string lives on the node's `config.sql` (query) or
`config.statements` (exec) or is generated by the plugin from validated
identifiers (upsert). Runtime values from upstream nodes flow ONLY
through bound parameters (`$1`, `$2`, …). There is no path by which
`inputs.params` can be string-interpolated into SQL — this is
injection-proof by construction, not by audit.

For upsert, the *identifiers* (table, columns, conflict targets) come
from `config` and are validated against the Postgres unquoted
identifier grammar (`[A-Za-z_][A-Za-z0-9_$]{0,62}`) and double-quoted
before splicing. The validator REJECTS quoted identifiers — accepting a
`"` from config is precisely the vector this whole pattern is designed
to close.

### 3. Connections are secrets referenced by name

The connection string is provided via `secrets.dsn` (a secret-ref to a
managed secret), not via `config`. `config.connection` is an
operator-facing label used for pool metrics / logs only. The same
pipeline points at dev vs prod by swapping the resolved secret per
environment — exactly the scoping ADR 0003 already standardised. An
inline DSN in a pipeline spec is impossible: the schema doesn't have a
field for one.

### 4. Shared, hash-keyed connection pool

A module-scoped `postgres-core` module owns a `Map<dsnHash, PoolEntry>`
cache. Two nodes that share a DSN share a pool — including across
worker tasks and synchronous-pipeline invocations on the API pod. The
prompt that motivated this work suggested keying by *connection name*,
but a name is a label chosen by the pipeline author; keying by the
resolved DSN means a single alias pointing at dev in one tenant and
prod in another keeps the two pools isolated, which is the property we
actually want. The label is retained on the pool entry for observability.

A graceful-shutdown hook (`SIGTERM` / `SIGINT` / `beforeExit`) ends
every cached pool. Tests substitute a fake driver via `__setPoolFactory`
so the unit suite stays offline.

### Categorisation in the existing contract

The actual `PluginManifest` shape (see ADR 0019) has no `sync_safe`
field and no per-plugin "MCP-exposable" flag — that's a *pipeline*-level
concern (`metadata.executionKind`, `metadata.mcpExpose`; see ADRs 0018,
0019). So the new plugins slot into the existing categories:

| Plugin            | Category | Sync-pipeline fit | MCP exposure (when in a pipeline) |
| ----------------- | -------- | ----------------- | --------------------------------- |
| `postgres_query`  | `tool`   | Yes — single round-trip, bounded by `maxRows`. | Safe; this is the intended retrieval shape for chat pipelines. |
| `postgres_upsert` | `sink`   | OK for small batches; primarily a batch-ingest plugin. | Not appropriate — write paths should not be MCP tools. |
| `postgres_exec`   | `tool`   | No — DDL is one-shot, not on the hot path. | Never. Operators should not flag a pipeline containing this node `mcpExpose: true`. |

`postgres_query` is categorised as `tool` rather than `retriever`
because the result shape is rows-of-records, not the embedding-derived
document set the `retriever` category implies for the Builder picker
and dataset-modality validator (ADR 0019). Pipelines that want to mix
rows into a rerank flow can wrap this node and an embedder behind
`merge_rrf` — that's a pipeline-author concern, not a plugin one.

`postgres_exec` is flagged with `capabilities: ["dangerous", "setup"]`
and is hard-gated by requiring `config.allowDDL === true` (the literal,
not a truthy value). The MCP exposure caveat is *operator policy*: the
contract does not, today, let a plugin declare itself "MCP-ineligible".
ADR 0021 proposes a first-class registry that would.

### Read-only by default

`postgres_query` opens a `BEGIN READ ONLY` transaction before running
the statement. This is the authoritative defence: even a misconfigured
SQL string with a write keyword would be rejected by Postgres. A
pre-flight statement-leading-keyword check (`assertLooksReadOnly`)
surfaces a clearer error before the round-trip but is intentionally
conservative — the txn is what we trust.

## Consequences

- Operators get a clean, secret-aware way to talk to their own
  Postgres from a pipeline. Migrations from ad-hoc query nodes to
  `postgres_query` are mechanical.
- Pool reuse keeps connection counts proportional to *operator
  configurations*, not to *pipeline runs* — critical for sync-pipeline
  latency.
- Identifier validation closes the one SQL-injection vector that
  parameter binding can't address.
- DDL is still possible (sometimes pipelines DO need to provision a
  table), but it's loud: `allowDDL: true` in the spec, `dangerous` in
  the manifest. Auditors can grep.
- This is the *category* ADR. The accompanying plugin (`postgres_query`
  / `_upsert` / `_exec`) and its tests live in `plugins/builtin-rag/`,
  matching the in-tree pattern. Future external-DB plugins (MySQL,
  ClickHouse, HTTP-as-DB) should reuse `postgres-core`'s pool
  abstraction or extract it into a shared package — see ADR 0021.

## Alternatives considered

1. **Add domain tables to Ragdoll's own DB** and let pipelines query
   them through a generic `db_query` plugin already pointed at
   `DATABASE_URL`. Tempting because there's no new connection
   plumbing. Rejected: ADR 0001 explicitly draws the boundary, ingest
   volume is unbounded and would degrade control-plane SLOs, and
   multi-tenant data isolation becomes a much harder schema problem
   than letting tenants own their own DB.

2. **One mega-plugin** `postgres_sql` that takes a free-form SQL +
   params and an `intent` flag (`query` vs `mutate` vs `ddl`). Compact
   but it dilutes the safety properties: the read-only-by-default
   posture, the identifier validator for upsert, and the DDL gate all
   want to be load-bearing structural decisions per plugin, not flags
   on one.

3. **Accept inline DSNs in pipeline config** for ergonomics. Rejected
   on principle: secrets ARE first-class in this system (ADR 0003) and
   we have no good answer for a DSN that ends up in a pipeline export
   and then in someone's clipboard.

4. **Key the pool cache by `config.connection` label** as the prompt
   suggested. Rejected because labels are author-chosen and the same
   label can resolve to different DSNs across environments — keying by
   the resolved DSN gives correct isolation. The label survives as
   metadata for observability.
